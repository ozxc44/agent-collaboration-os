# Agent Collaboration OS — 产品规划

> 状态：2026-06-29 审计后撰写。基于代码实际状态（33k 行后端、197 API、37 实体、统一 runtime），不是设想。

## 一、这个产品现在到底是什么

经过这次端到端验证，平台的**真实能力边界**已经清晰：

### 已经跑通的核心闭环
```
PM（主 agent）派活 → 平台路由 → 本地模型实例化（kimi/mimo/claude/hermes/codex/deepseek）
→ agent 用完整能力干活（读文件/写代码/用工具）→ 提交变更集 → PM 审核 → 合并
```

### 独有的差异化（不是"又一个 agent 框架"）
1. **统一本地模型 runtime**——一个进程服务一台机器上所有本地模型（kimi/mimo/claude/hermes/codex），按 agent_id 精确路由，每个模型一个 agent 身份。这是**别人没有的**——LangGraph/CrewAI 是编排库，不管"模型在哪、怎么实例化"。
2. **真实 Git 后端**（isomorphic-git）+ Gitea 网关——不是数据库模拟的 git，是真正的版本控制。
3. **PM/worker 编排 + 变更集审核**——类 GitHub PR 的多 agent 协作，带验收。
4. **MD 驱动工作流**（goal.md → TASK.md → RESULT.md → REVIEW.md）——人机可读的协作契约。

### 现在的真实短板（诚实评估）
- **前端是原生 HTML**（33k 行，无框架），不适合商业化产品形态
- **单租户**——目前是单实例部署，没有多租户隔离
- **认证简单**——JWT + agent key，没有 OAuth/SSO/团队管理
- **无计费/配额**——没有按用量计费的基础设施
- **运维未产品化**——靠 docker-compose 手动部署，没有托管/一键部署
- **文档偏开发笔记**，没有面向用户的文档

## 二、商业化 vs 开源——我的判断

### 结论先行：**开源为主 + 商业托管（open-core）**

理由：

### 为什么不该纯商业化
1. **市场拥挤**——Devin/Factory/Cursor/Sourcegraph 都在做"AI 写代码"。一个独立项目纯商业化很难打。
2. **获客成本高**——B2B 开发者工具的获客极贵，需要销售团队。
3. **信任问题**——闭源的"AI agent 平台"难以让企业把代码托付给它。
4. **生态依赖**——这个项目依赖 kimi/mimo/claude 等本地模型生态，纯闭源会和生态对立。

### 为什么不该纯开源（无商业）
1. **长期维护需要钱**——33k 行代码 + 197 API 的维护是全职工作量。
2. **企业需要托管**——大多数企业不想自己部署，愿意为托管付费。
3. **纯开源项目容易死**——没有收入支撑，维护者会倦怠。

### 最佳路径：open-core（开源核心 + 商业托管）
```
开源（GitHub）              商业（托管 SaaS）
─────────────              ──────────────
核心 runtime + 编排引擎     托管平台（不用自己部署）
统一模型路由                多租户 + 团队管理
本地 agent 接入             企业 SSO + 审计
CLI + SDK                  计费 + 配额
社区贡献                   优先支持 + SLA
```

**参照模式**：Supabase、GitLab、Sentry、Posthog——都是开源核心 + 商业托管，已被验证可行。

## 三、产品定位（一句话）

> **让任意本地 AI 模型（kimi/claude/codex/...）自动协作完成软件项目，像 GitHub 团队一样派活、干活、审核、合并。**

不是"又一个 AI 编程助手"（那是 Cursor/Copilot 的赛道），而是**多 agent 协作平台**——治理一群 AI agent 像治理一个开发团队。

### 目标用户
- **AI-native 开发团队**——已经有多个 AI agent（claude/codex/kimi），需要让它们协作而不是各自为战
- **AI 自动化极客**——想让一组 agent 自动完成整个项目（需求→开发→审核→合并）
- **企业 AI 平台团队**——需要在内部托管一套"AI agent 协作治理"平台

### 阶段 1：开源发布 + 社区（2-3 个月）
**目标**：建立用户基础和反馈循环。

- [ ] 开源核心（runtime + 编排引擎 + Git 后端 + CLI）
- [ ] demo 视频："5 个 AI agent 协作完成一个真实项目"

- [ ] 社区贡献指南（如何加一个新的本地模型后端）

### 阶段 2：商业托管 MVP（3-6 个月）
**目标**：验证有人愿意为托管付费。

- [ ] 多租户改造（隔离不同团队的项目/agent）

### 阶段 3：企业版（6-12 个月）
- [ ] 私有部署支持（企业内网）
- [ ] 合规（SOC2 / 数据驻留）

- [ ] 定制集成（Jira/Linear/Slack/GitHub）

## 五、现在的核心决策点（需要你定）

### 决策 1：品牌名
"zhuzeyang-agent" 必须改。建议候选：
- **AgentOS** / **AgentForge**（协作平台感）
- **Hive** / **Swarm**（多 agent 群体感）
- **Crew Hub** / **Dock**（团队协作感）

### 决策 2：开源协议
- **MIT/Apache**——最开放，利于采纳，但别人可以闭源 fork
- **AGPL**——强制开源派生，保护商业（GitLab/Supabase 用这个）

### 决策 3：商业托管 vs 私有部署优先
- 托管优先：快速验证付费意愿，但要做多租户
- 私有优先：企业更容易接受（数据不出内网），但单价高、周期长

### 决策 4：是否保留"桌面端接入"（codex/zcode）
你说要保留——我同意。**桌面端 agent（codex app / zcode）是差异化**：大多数 agent 平台只支持 API 模型，我们支持本地桌面 agent 实例化。这是护城河，应该作为主打特性。

## 六、风险与诚实评估

### 最大的风险
1. **市场时机**——如果 Devin/Factory 之类先做出"多 agent 协作"，我们的窗口关闭
2. **维护负担**——一个人维护 33k 行很难，需要团队或社区
3. **模型生态变化**——kimi/mimo/claude 的 CLI 接口会变，runtime 要持续适配
4. **商业化验证**——不确定是否真有人为"多 agent 协作托管"付费

### 诚实的不确定
- 我**不能确定**这个能赚到钱。多 agent 协作是热点，但"愿意付费"和"觉得酷"是两回事。
- 开源能积累声誉和用户，但**不一定能转化成收入**。
- 最大价值可能是**被收购**——做出有特色的 agent 协作平台，被大厂（如 Anthropic/OpenAI/字节）收购。这也是合法的退出路径。

### 我的建议
1. **先开源**（阶段 0+1），用社区验证产品价值——成本最低
2. **观察 3 个月**，看是否有真实用户和 stars 增长
3. **再决定商业化**——有 traction 再做托管；没 traction 就保持开源项目

不要在没验证需求前投钱做托管——那是最大的浪费。
