## 一、调研背景与方法

通过平台 PM 向三个本地模型分别派发独立调研任务：
- **Kimi**：多 agent 协作平台市场与竞品分析
- **MiMo**：商业化路径分析（open-core / SaaS / 被收购）
- **Codex**：技术债评估与开源准备清单

三个模型各自推理产出，PM 回收整合。以下为整合后的最终结论。

---

## 二、市场定位（Kimi 分析）

### 竞品格局

| 竞品 | 定位 | 商业模式 | 开源 |
|------|------|---------|------|
| **LangGraph** | 图状状态机编排，checkpoint/分支/人环 | LangSmith 按席位 $39/月起 | MIT |
| **CrewAI** | 角色化多 agent（role/goal/backstory） | AOP/SaaS $25/月起 | MIT |
| **AutoGen/MS** | 对话式多 agent，Azure/.NET 生态 | Azure AI Foundry | MIT |
| **Devin** | 端到端自主软件工程师（云端 VM） | 闭源 SaaS，$2.25/ACU | 闭源 |
| **OpenHands** | 开源云 coding agent，模型无关 | 开源+企业服务 | MIT |

### 我们的差异化机会（三个模型共识）

1. **私有化与合规**——Devin/Azure 数据出境，金融/医疗/政企需要本地 runtime（**这是最大差异化**）
2. **工程化 PM 编排**——现有框架是对话式/图式，缺少真实软件项目的 PM 拆解、任务分派、worker 并行、冲突合并、审计追踪（**我们的 PM/worker 编排 + 变更集审核正是这个**）
3. **模型中立 + 多模型路由**——统一 runtime 同时调度本地小模型与远程大模型（**我们的统一 runtime 独有**）
4. **成本可控**——避免按 ACU/节点计费，本地模型统一调度降低推理成本

**定位结论**：面向中大型企业的"私有化、可审计、PM-worker 分层"的工程 Agent 平台，填补开源框架（需自研）与闭云 SaaS（成本高、数据外泄）之间的空白。

## 三、商业化路径（MiMo 分析）

| 路径 | 可行性 | 时间 | 风险 |
|------|--------|------|------|
| **Open-Core**（推荐）| **高** | 12-18 月达 $1M ARR | 社区分裂；开源/闭源边界 |
| 纯 SaaS | 中低 | 6-12 月上线，天花板低 | 获客贵；大厂碾压 |
| 被收购 | 中 | 18-36 月，看差异化 | 估值受市场波动 |

### MiMo 推荐（PM 认同）
> **Open-Core 为主，被收购为退出备选。**
>
> Agent 协作平台的核心价值在生态粘性——开源建开发者社区，企业版变现，保持独立性的同时保留被收购的选项价值。纯 SaaS 不推荐：用户偏技术型，强绑定云会降低采用，且直面 OpenAI/Anthropic 官方平台竞争。

## 四、开源前必做（Codex 分析，优先级排序）

### 必做（开源前）
1. **安全清理**——密钥、内网地址（NAS IP/密码）、测试账号、私有部署脚本脱敏（**最优先**）
2. **架构边界说明**——明确单租户假设、API 分层、核心数据模型、agent 生命周期
3. **最小可运行路径**——Docker Compose 一键起、种子数据、健康检查、关键 smoke test
4. **API 收敛与契约化**——补 OpenAPI/接口清单，标注废弃/内部 API，防兼容债爆炸
5. **前端重构到可维护基线**——不必完美，但组件化、路由清晰，停止原生 HTML 扩散

### 可后做
多租户、计费、细粒度权限、企业 SSO、审计报表、UI 精修、大规模性能优化。

## 五、整合最终方案（PM 决策）

### 战略：Open-Core，开源先行验证

**三模型共识 + PM 判断**：

1. **开源核心**（runtime + 编排引擎 + Git 后端 + CLI + SDK）——MIT 或 Apache
2. **商业托管**（多租户 + SSO + 计费 + SLA）——open-core 模式
3. **被收购**作为退出选项保留

### 差异化护城河（主打）
- **统一本地模型 runtime**（kimi/mimo/claude/codex/hermes 一键接入，精确路由）——竞品没有
- **PM/worker 工程化编排 + 变更集审核**——比 LangGraph/CrewAI 的对话式编排更贴近真实软件开发
- **私有化部署**——企业数据不出内网，对抗 Devin/Azure 的数据出境问题

### 执行顺序（综合 Codex 技术债 + MiMo 商业判断）

**阶段 0（1-2 月）：开源准备**
- 安全清理（去 NAS IP/密码/个人 key）← Codex 第一优先
- 定品牌名 + 协议（MIT 推荐，最利采纳）
- 前端重构基线（组件化，停 HTML 扩散）
- Docker Compose 一键部署 + demo
- 面向用户文档

**阶段 1（2-3 月）：开源发布**
- GitHub 公开 + demo 视频（5 agent 协作的视觉冲击）
- 接入主流模型生态（kimi/claude/codex 官方支持）
- 社区（Discord/Slack）

**阶段 2（3-6 月）：商业托管 MVP**
- 多租户 + 团队 RBAC + SSO
- 托管平台（注册即用）
- 用量计费

**阶段 3（6-12 月）：企业版**
- 私有部署支持（内网）
- 合规（SOC2）
- SLA + 定制集成

### 关键决策（需立即定）
1. **品牌名**——"zhuzeyang-agent" 必须改
2. **协议**——MIT（最开放）vs AGPL（保护商业）
3. **托管 vs 私有优先**——建议托管优先（快速验证付费）

## 六、诚实的不确定

- **不能保证赚钱**——多 agent 协作是热点，但"愿意付费"和"觉得酷"是两回事
- **最大价值可能是被收购**——做出有特色的 agent 协作平台，被 Anthropic/OpenAI/字节收购
- **建议先开源验证 3 个月**——有 traction（stars/用户）再做托管；没 traction 保持开源项目
- **不要在没验证需求前投钱做托管**——最大浪费
